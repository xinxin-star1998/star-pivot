# 代码审查流程规范

## 1. 概述

本文档定义了 StarPivot 项目的代码审查流程，旨在确保代码质量、促进知识共享、减少缺陷并提高团队协作效率。

## 2. 审查原则

### 2.1 核心原则
- **质量优先**：确保代码符合项目规范和质量标准
- **建设性反馈**：提供建设性的、可操作的建议
- **及时响应**：在 24 小时内完成代码审查
- **相互尊重**：保持专业和尊重的态度
- **持续改进**：从每次审查中学习和改进

### 2.2 审查范围
- 代码逻辑和算法
- 代码风格和格式
- 错误处理和边界情况
- 性能和安全问题
- 测试覆盖率和质量
- 文档完整性

## 3. 审查流程

### 3.1 提交前准备

#### 3.1.1 开发者职责
```bash
# 1. 运行代码检查
mvn clean compile spotless:check -pl star-pivot-controller -am
cd star-pivot-ui && npm run lint && npm run lint:prettier && npm run lint:stylelint

# 2. 运行测试
mvn test -pl star-pivot-module -am
cd star-pivot-ui && npm run test

# 3. 确保构建成功
mvn clean package -DskipTests
cd star-pivot-ui && npm run build
```

#### 3.1.2 提交规范
- 遵循 Conventional Commits 规范
- 提交信息格式：`<type>(<scope>): <subject>`
- 类型：feat, fix, docs, style, refactor, test, chore
- 示例：
  ```
  feat(auth): add JWT token refresh mechanism
  fix(user): resolve role permission caching issue
  docs(readme): update deployment instructions
  ```

### 3.2 创建 Pull Request

#### 3.2.1 PR 标题格式
```
[类型] 简短描述
```
示例：
```
[FEAT] Add user import/export functionality
[FIX] Resolve login session timeout issue
[REFACTOR] Optimize database query performance
```

#### 3.2.2 PR 描述模板
```markdown
## 变更说明
简要描述本次变更的内容和目的

## 变更类型
- [ ] 新功能 (feature)
- [ ] Bug 修复 (bugfix)
- [ ] 代码重构 (refactor)
- [ ] 文档更新 (documentation)
- [ ] 性能优化 (performance)
- [ ] 测试相关 (testing)

## 影响范围
- [ ] 后端 (star-pivot-controller)
- [ ] 前端 (star-pivot-ui)
- [ ] 框架层 (star-pivot-framework)
- [ ] 业务层 (star-pivot-module)

## 测试情况
- [ ] 单元测试已通过
- [ ] 集成测试已通过
- [ ] 手动测试已完成
- [ ] 测试覆盖率达标

## 检查清单
- [ ] 代码符合项目规范
- [ ] 已添加必要的注释
- [ ] 已更新相关文档
- [ ] 无安全漏洞
- [ ] 无性能问题

## 相关 Issue
Closes #(issue number)

## 截图/演示
(如有 UI 变更，请提供截图或演示视频)
```

### 3.3 代码审查

#### 3.3.1 审查者职责
1. **自动检查通过**：确保 CI/CD 流程全部通过
2. **代码质量检查**：
   - 代码逻辑正确性
   - 代码风格一致性
   - 错误处理完整性
   - 性能和安全问题

3. **测试检查**：
   - 单元测试覆盖率
   - 测试用例完整性
   - 边界情况处理

4. **文档检查**：
   - 代码注释充分
   - API 文档更新
   - README 更新

#### 3.3.2 审查标准

##### 代码质量标准
| 指标 | 标准 | 说明 |
|------|------|------|
| 代码覆盖率 | ≥ 70% | 新增代码必须达到 |
| 圈复杂度 | ≤ 10 | 单个方法复杂度 |
| 代码重复率 | ≤ 3% | 重复代码比例 |
| SonarQube 评分 | A | 质量门禁要求 |
| 安全漏洞 | 0 | 严重和高危漏洞 |

##### 审查要点
- ✅ 代码逻辑清晰，易于理解
- ✅ 遵循项目代码规范
- ✅ 错误处理完善
- ✅ 无安全漏洞
- ✅ 性能合理
- ✅ 测试充分
- ✅ 文档完整

##### 常见问题
- ❌ 硬编码配置信息
- ❌ 未处理的异常
- ❌ SQL 注入风险
- ❌ 资源未释放
- ❌ 缺少单元测试
- ❌ 代码重复

### 3.4 反馈与修改

#### 3.4.1 反馈格式
```markdown
### 问题类型
- [ ] 阻塞性问题 (必须修复)
- [ ] 严重问题 (建议修复)
- [ ] 改进建议 (可选)

### 问题描述
具体描述问题所在

### 建议方案
提供具体的修改建议

### 代码示例
```java
// 当前代码
public void method() {
    // ...
}

// 建议修改
public void method() {
    // ...
}
```
```

#### 3.4.2 响应时间要求
- **开发者**：收到反馈后 48 小时内响应
- **审查者**：收到修改后 24 小时内再次审查

### 3.5 合并

#### 3.5.1 合并条件
- ✅ 所有 CI/CD 检查通过
- ✅ 至少 1 名审查者批准
- ✅ 所有审查意见已解决
- ✅ 无冲突或冲突已解决
- ✅ 代码符合质量标准

#### 3.5.2 合并方式
- **Squash and merge**：推荐，保持提交历史清晰
- **Rebase and merge**：适用于需要保留详细提交历史的情况
- **Merge commit**：不推荐，除非有特殊需求

## 4. 质量门禁

### 4.1 自动化检查
- ✅ 编译成功
- ✅ 所有测试通过
- ✅ 代码覆盖率达标
- ✅ 静态代码分析通过
- ✅ 安全扫描无高危漏洞
- ✅ 代码格式检查通过

### 4.2 SonarQube 质量门禁
- 代码覆盖率 ≥ 70%
- 新增代码覆盖率 ≥ 80%
- 重复率 ≤ 3%
- 可靠性评级 A
- 安全性评级 A
- 可维护性评级 A
- 无阻断性问题

## 5. 分支策略

### 5.1 分支命名规范
- `main/master`：生产环境分支
- `develop`：开发环境分支
- `feature/xxx`：功能开发分支
- `bugfix/xxx`：Bug 修复分支
- `hotfix/xxx`：紧急修复分支
- `release/xxx`：发布分支

### 5.2 工作流程
```
1. 从 develop 创建 feature/xxx 分支
2. 在 feature/xxx 分支开发
3. 创建 PR 到 develop 分支
4. 代码审查通过后合并到 develop
5. 从 develop 创建 release/xxx 分支
6. 测试通过后合并到 main/master
```

## 7. 最佳实践

### 7.1 开发者最佳实践
1. **小步提交**：保持 PR 规模适中（建议 < 500 行）
2. **自测先行**：提交前充分自测
3. **文档同步**：及时更新相关文档
4. **响应及时**：快速响应审查意见

### 7.2 审查者最佳实践
1. **及时审查**：在 24 小时内完成审查
2. **建设性反馈**：提供具体、可操作的建议
3. **关注重点**：优先关注逻辑、安全、性能问题
4. **知识共享**：通过审查分享最佳实践

### 7.3 团队协作
1. **定期回顾**：每月回顾代码审查流程
2. **持续改进**：根据反馈优化流程
3. **知识分享**：定期组织代码审查分享会
4. **工具优化**：持续改进自动化工具

## 8. 常见问题

### Q1: PR 多久会被审查？
A: 通常在 24 小时内完成审查，如超过 48 小时未审查，可在团队群提醒。

### Q2: 审查意见不同意怎么办？
A: 可以在 PR 中讨论，如无法达成一致，可邀请技术负责人仲裁。

### Q3: 如何处理紧急修复？
A: 紧急修复可以简化审查流程，但必须经过至少 1 名审查者批准。

### Q4: 测试覆盖率不够怎么办？
A: 新增代码覆盖率必须达到 70%，如确实难以达到，需要在 PR 中说明原因并获得批准。

### Q5: 如何处理遗留代码？
A: 遗留代码可以在后续 PR 中逐步重构，本次 PR 只关注新增代码质量。
