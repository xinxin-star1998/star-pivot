# 权限编码命名规范与数据权限规范

> 本文用于统一 StarPivot 项目的**权限编码命名规则**、**核心权限清单**以及**数据权限（DataScope）使用约定**，并给出可选的 MyBatis 拦截器扩展方案，便于后续演进。  
> JWT 鉴权、匿名放行扩展见 [`star-pivot-security-使用说明.md`](star-pivot-security-使用说明.md)。

---

## 一、权限编码命名规范（模块:资源:操作）

### 1. 命名总体规则

- 基本格式：`模块:资源:操作`
  - **模块**：系统一级域（如 `system`、`monitor`、`job` 等）
  - **资源**：具体资源对象（如 `menu`、`user`、`role`、`dict` 等）
  - **操作**：动词或操作类型（只用小写英文单词，必要时用中划线连接）
- 命名风格：
  - 统一使用小写字母和中划线（如确需多词组合）
  - 严禁中英文混杂、缩写不明的编码（如 `sys:m:add1` 等）
  - 避免同义词混用：**统一使用固定动词集合**

### 2. 推荐操作动词集合

- **查询类**：`query`（列表 / 详情均可，根据接口语义区分）
- **新增类**：`add`
- **修改类**：`edit`
- **删除类**：`delete`
- **导入导出**：`import`、`export`
- **状态变更**：`enable`、`disable`、`reset-pwd` 等
- **分配关系**：`assign`（如 `system:role:assign-user`）

> 约定：**同一资源上不要混用 `remove` / `delete` 等同义词**，统一选择一种（本项目统一使用 `delete`）。

### 3. 示例

- 菜单管理（system:menu）
  - `system:menu:query`：菜单查询（含树）
  - `system:menu:add`：新增菜单 / 权限
  - `system:menu:edit`：编辑菜单 / 权限
  - `system:menu:delete`：删除菜单 / 权限
- 用户与角色
  - `system:user:query`：查询用户列表 / 详情
  - `system:user:add`：新增用户
  - `system:user:edit`：编辑用户
  - `system:user:delete`：删除用户
  - `system:user:import`：导入用户（Excel 批量导入）
  - `system:user:export`：导出用户（Excel 导出）
  - `system:role:assign-user`：给角色分配用户
- 路由与权限（预留）
  - `system:router:query`：查询动态路由 / 用户路由

> 前后端在新增权限点时，**必须同步更新本规范或补充清单**，并保持 Controller 注解、菜单 `perms` 字段、前端按钮 `hasAuth(...)` / `v-auth` 一致。

## 二、当前核心权限清单（节选）

> 以下为当前代码中已经使用且经过梳理的核心权限点，后续新增请参照同一命名风格。

### 1. 菜单管理（SysMenuController / 菜单页面）

- `system:menu:query`
  - 用途：菜单树查询、单个菜单详情查询、上级菜单列表查询。
  - 后端：`SysMenuController.menuTree` / `getParent` / `getById` 上使用 `@PreAuthorize("hasAuthority('system:menu:query')")`
  - 前端：菜单管理页进入和数据加载依赖该权限。

- `system:menu:add`
  - 用途：新增菜单、为菜单新增权限按钮。
  - 后端：`SysMenuController.add`
  - 前端：菜单管理页“添加菜单”“新增权限”按钮显示控制。

- `system:menu:edit`
  - 用途：编辑菜单 / 权限按钮。
  - 后端：`SysMenuController.edit`
  - 前端：菜单/权限行的“编辑”操作按钮显示控制。

- `system:menu:delete`
  - 用途：删除菜单 / 权限按钮。
  - 后端：`SysMenuController.remove`
  - 前端：
    - 权限按钮删除：`hasAuth('system:menu:delete')`
    - 菜单删除按钮：`hasAuth('system:menu:delete')`

> 建议：后续如增加“菜单导入/导出”等操作，对应新增 `system:menu:import` / `system:menu:export`，并在此处补充说明。

### 2. 用户管理（SysUserController / 用户管理页面）

- `system:user:query`
  - 用途：查询用户列表、用户详情。
  - 后端：`SysUserController.pageList` / `getUserById` 上使用 `@PreAuthorize("hasAuthority('system:user:query')")`
  - 前端：用户管理页进入和数据加载依赖该权限。

- `system:user:add`
  - 用途：新增用户。
  - 后端：`SysUserController.addUser`
  - 前端：用户管理页“添加用户”按钮显示控制。

- `system:user:edit`
  - 用途：编辑用户、重置密码、启用/禁用用户。
  - 后端：`SysUserController.updateUser` / `resetPwd` / `changeStatus`
  - 前端：用户行的“编辑”“重置密码”“启用/禁用”操作按钮显示控制。

- `system:user:delete`
  - 用途：删除用户（支持批量删除）。
  - 后端：`SysUserController.remove`
  - 前端：用户行的“删除”操作按钮显示控制。

- `system:user:import`
  - 用途：导入用户（EasyExcel 批量导入）、下载导入模板。
  - 后端：`SysUserController` — `POST /api/sys/user/import`、`GET /api/sys/user/importTemplate`，`@PreAuthorize("hasAuthority('system:user:import')")`
  - 前端：用户管理页「导入用户」及导入弹窗内「下载模板」（`v-auth="'system:user:import'"` + `ExcelImportDialog`）。

- `system:user:export`
  - 用途：导出用户（EasyExcel）。
  - 后端：`SysUserController` — `POST /api/sys/user/export`，`@PreAuthorize("hasAuthority('system:user:export')")`
  - 前端：用户管理页「导出用户」（`v-auth="'system:user:export'"`）。

> **模块化导入导出与权限**（实现见 [`通用导入导出使用说明.md`](通用导入导出使用说明.md)）：
> - 各业务在**自身 Controller** 暴露 `/export`、`/import`、`/importTemplate`，权限写在对应方法上，**不再**使用已移除的统一入口 `/common/import-export/*`。
> - 权限编码仍为 `模块:资源:import` / `模块:资源:export`，与菜单 `perms`、前端 `v-auth` 一致。
> - 示例：`system:config:export`（`SysConfigController.export`）、`mall:base:import` / `mall:sale:import`（`PmsAttrController`，按 `attrType` 区分业务）、`mall:group:import` / `mall:group:export`（`PmsAttrGroupController`）。
> - 新增业务的导入导出：先在菜单配置 `import` / `export` 权限点，再在 Controller 三接口上使用 `@PreAuthorize`。

### 3. 预留与扩展

- `system:router:query`（预留）
  - 如将来需要更细粒度控制获取动态路由 / 用户菜单树，可在 `RouterController` 上使用。

## 三、数据权限（DataScope）使用规范

### 1. 数据范围类型与优先级

在 `DataScope` / `DataScopeService` 中统一定义数据范围：

- 全部数据：`ALL`
- 自定义部门：`CUSTOM`
- 本部门：`DEPT`
- 本部门及以下：`DEPT_AND_CHILD`
- 仅本人：`SELF`

优先级从大到小约定为：

> `ALL(1) > CUSTOM(2) > DEPT(3) > DEPT_AND_CHILD(4) > SELF(5)`

注意：

- 任何涉及调整优先级或含义的改动，必须同步更新：
  - `DataScopeService` 内部优先级映射和注释
  - 涉及 DataScope 的说明文档（如本文件和 `权限问题分析报告.md`）

### 2. Service 层使用约定

- 获取当前用户数据权限：
  - 统一通过 `DataScopeService.getCurrentUserDataScope()`（或对应封装）获取。
- 组装参数传给 Mapper：
  - `dataScope`：完整的 DataScope 对象
  - `deptIds`：可见部门 ID 列表（如适用）
  - `userDeptId`：当前用户部门 ID（如适用）
  - `userId`：当前用户 ID（用于“仅本人”场景）
- 示例（伪代码）：

```java
DataScope dataScope = dataScopeService.getCurrentUserDataScope();
Map<String, Object> param = new HashMap<>();
param.put("dataScope", dataScope);
param.put("deptIds", dataScope.getDeptIds());
param.put("userDeptId", dataScope.getUserDeptId());
param.put("userId", dataScope.getUserId());
param.put("userReqBo", userReqBo);
IPage<SysUser> pageList = sysUserMapper.selectPageList(page, param);
```

### 3. Mapper 层 XML 约定

- 对用户相关查询（分页、角色已分配 / 未分配列表等），统一在 `WHERE` 中使用公共 SQL 片段：

```xml
<!-- SysUserMapper.xml 中统一的数据权限过滤片段 -->
<sql id="dataScopeFilter">
    <!-- 数据权限过滤：使用choose标签避免SQL注入风险 -->
    <choose>
        <when test="param.dataScope != null and param.dataScope.sqlFilter == '1=1'">
            <!-- 全部数据，不需要额外条件 -->
        </when>
        <when test="param.dataScope != null and param.dataScope.sqlFilter == '1=0'">
            AND 1=0
        </when>
        <when test="param.dataScope != null and param.deptIds != null and param.deptIds.size() > 0">
            AND u.dept_id IN
            <foreach collection="param.deptIds" item="deptId" open="(" separator="," close=")">
                #{deptId}
            </foreach>
        </when>
        <when test="param.userDeptId != null">
            AND u.dept_id = #{param.userDeptId}
        </when>
        <when test="param.userId != null">
            AND u.create_by = #{param.userId}
        </when>
    </choose>
</sql>
```

- 各查询通过 `<include refid="dataScopeFilter"/>` 引用，避免重复书写：

```xml
<where>
    u.del_flag = '0'
    <!-- 业务条件若干 -->
    <include refid="dataScopeFilter"/>
</where>
```

> 约定：**严禁在 XML 中直接使用 `${}` 拼接数据权限 SQL 片段**，统一通过 `<choose>` + 绑定参数实现，降低 SQL 注入风险。

## 四、MyBatis 拦截器统一数据权限（可选方案）

> 本节为中长期优化方向，目前以文档设计为主，具体落地可按需实现。

### 1. 适用场景

- 项目中数据权限使用场景增多、横跨多个 Mapper，且 XML 中反复出现类似 `dataScopeFilter` 片段。
- 需要在不改动大量 XML 的前提下，统一给指定表 / 别名追加数据权限条件。

### 2. 设计思路

- 使用 MyBatis 拦截器（如实现 `org.apache.ibatis.plugin.Interceptor` 或 MyBatis Plus 的 `InnerInterceptor`）：
  - 拦截 `Executor.query(...)` 或 SQL 解析过程。
  - 根据 Mapper 方法 / 自定义注解 / 约定，将数据权限条件注入到原始 SQL 中。
- 典型做法：
  1. 在 Service 或 Mapper 方法上增加注解，如：
     - `@DataScope(tableAlias = "u")`
  2. 拦截器在执行前解析注解 + 当前登录用户信息，构建 DataScope。
  3. 对原 SQL 进行 AST 级解析（或简单字符串拼接，需谨慎），将条件追加到 `WHERE` 部分。

### 3. 伪代码示例（仅示意）

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface DataScopeInject {
    String tableAlias() default "u";
}

@Intercepts({
    @Signature(
        type = Executor.class,
        method = "query",
        args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class}
    )
})
public class DataScopeInterceptor implements Interceptor {

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        MappedStatement ms = (MappedStatement) invocation.getArgs()[0];
        Object parameter = invocation.getArgs()[1];

        // 1. 根据 MappedStatementId 找到对应 Mapper 方法与 @DataScopeInject 注解
        // 2. 若无注解则直接放行
        // 3. 构建当前用户 DataScope，对原 SQL 追加过滤条件
        // 4. 将修改后的 BoundSql/SqlSource 重新设置回 MappedStatement

        return invocation.proceed();
    }
}
```

### 4. 使用建议

- 短期内可继续使用当前 XML 片段方式（已通过公共 `<sql>` 片段减少重复）。
- 当：
  - 数据权限涉及的表显著增多，或
  - 需要对多种别名 / 多表 JOIN 做统一处理
  
  时，再落地上述拦截器方案，并在此文档基础上补充具体实现与注意事项。

## 五、维护与变更约定

1. 新增权限点时：
   - 先在本规范中确定命名（`模块:资源:操作`），再在后端注解 / 菜单 `perms` / 前端 `hasAuth` / `v-auth` 中使用。
   - 若涉及 Excel 导入导出，同步查阅 [`通用导入导出使用说明.md`](通用导入导出使用说明.md) 配置 Controller 三接口。
2. 调整数据权限逻辑时：
   - 必须同时更新：
     - `DataScope` / `DataScopeService` 注释
     - 相关 Mapper XML 中的公共片段（或拦截器实现）
     - 本文档对应章节。
3. 若引入 MyBatis 拦截器：
   - 在本文件新增“实现细节”章节，并说明与现有 XML 方式的兼容策略（如逐步迁移或并行支持）。
