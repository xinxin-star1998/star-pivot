# 代码质量门禁配置

## 1. 概述

本文档定义了 StarPivot 项目的代码质量门禁标准，所有代码变更必须通过这些质量检查才能合并到主分支。

## 2. 质量门禁标准

### 2.1 后端质量标准

#### 2.1.1 代码覆盖率
```xml
<!-- pom.xml JaCoCo 配置 -->
<rules>
    <rule>
        <element>BUNDLE</element>
        <limits>
            <limit>
                <counter>INSTRUCTION</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.50</minimum>
            </limit>
            <limit>
                <counter>BRANCH</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.40</minimum>
            </limit>
            <limit>
                <counter>LINE</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.50</minimum>
            </limit>
            <limit>
                <counter>METHOD</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.50</minimum>
            </limit>
            <limit>
                <counter>CLASS</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.50</minimum>
            </limit>
        </limits>
    </rule>
</rules>
```

#### 2.1.2 SonarQube 质量指标

| 指标 | 标准 | 说明 |
|------|------|------|
| 代码覆盖率 | ≥ 50% | 整体代码覆盖率 |
| 新增代码覆盖率 | ≥ 70% | 新增代码必须达到 |
| 重复率 | ≤ 3% | 代码重复比例 |
| 圈复杂度 | ≤ 10 | 单个方法复杂度 |
| 认知复杂度 | ≤ 15 | 方法认知复杂度 |
| 文件行数 | ≤ 500 | 单个文件行数 |
| 方法行数 | ≤ 50 | 单个方法行数 |
| 参数数量 | ≤ 7 | 方法参数数量 |

#### 2.1.3 质量等级要求
- **可靠性**: A 级（无阻断性问题）
- **安全性**: A 级（无高危漏洞）
- **可维护性**: A 级（技术债务可控）
- **覆盖率**: B 级以上

## 3. 自动化检查

### 3.1 CI/CD 检查项

#### 3.1.1 后端检查
```yaml
# .github/workflows/ci-cd.yml
- 编译检查
  - mvn clean compile
  - mvn spotless:check

- 测试检查
  - mvn test
  - mvn jacoco:report

- 质量检查
  - SonarQube 扫描
  - 代码覆盖率检查
  - 安全漏洞扫描
```

#### 3.1.2 前端检查
```yaml
# .github/workflows/ci-cd.yml
- 代码检查
  - npm run lint
  - npm run lint:prettier
  - npm run lint:stylelint

- 类型检查
  - npx vue-tsc --noEmit

- 测试检查
  - npm run test
  - npm run test:coverage

- 构建检查
  - npm run build
```

### 3.2 质量门禁配置

#### 3.2.1 SonarQube 质量门禁
```properties
# sonar-project.properties
sonar.qualitygate.wait=true

# 质量门禁规则
# 1. 代码覆盖率 ≥ 50%
# 2. 新增代码覆盖率 ≥ 70%
# 3. 重复率 ≤ 3%
# 4. 可靠性评级 A
# 5. 安全性评级 A
# 6. 可维护性评级 A
# 7. 无阻断性问题
```

#### 3.2.2 GitHub Actions 质量门禁
```yaml
# .github/workflows/ci-cd.yml
quality-gate:
  name: Quality Gate
  runs-on: ubuntu-latest
  needs: [backend-quality, frontend-quality, security-scan]
  
  steps:
    - name: Check quality gate status
      run: |
        if [ "${{ needs.backend-quality.result }}" != "success" ] || \
           [ "${{ needs.frontend-quality.result }}" != "success" ]; then
          echo "Quality gate failed!"
          exit 1
        fi
```

## 4. 本地开发检查

### 4.1 Pre-commit Hooks

#### 4.1.1 Husky 配置
```bash
# 安装 Husky
npm install -D husky

# 初始化
npx husky install

# 添加 pre-commit hook
npx husky add .husky/pre-commit "npm run lint && npm run test"
```

#### 4.1.2 Pre-commit 脚本
```bash
#!/bin/sh
. "$(dirname "$0")/_/husky.sh"

echo "Running pre-commit checks..."

# 后端检查
echo "Checking backend code..."
mvn clean compile spotless:check -pl star-pivot-controller -am
if [ $? -ne 0 ]; then
  echo "Backend check failed!"
  exit 1
fi

# 前端检查
echo "Checking frontend code..."
cd star-pivot-ui
npm run lint
if [ $? -ne 0 ]; then
  echo "Frontend lint failed!"
  exit 1
fi

npm run lint:prettier
if [ $? -ne 0 ]; then
  echo "Prettier check failed!"
  exit 1
fi

npm run lint:stylelint
if [ $? -ne 0 ]; then
  echo "Stylelint check failed!"
  exit 1
fi

# 运行测试
echo "Running tests..."
npm run test
if [ $? -ne 0 ]; then
  echo "Tests failed!"
  exit 1
fi

echo "All checks passed!"
```

### 4.2 Pre-push Hooks

#### 4.2.1 Pre-push 脚本
```bash
#!/bin/sh
. "$(dirname "$0")/_/husky.sh"

echo "Running pre-push checks..."

# 运行完整测试套件
echo "Running full test suite..."
mvn test -pl star-pivot-module -am
if [ $? -ne 0 ]; then
  echo "Backend tests failed!"
  exit 1
fi

cd star-pivot-ui
npm run test:coverage
if [ $? -ne 0 ]; then
  echo "Frontend tests failed!"
  exit 1
fi

# 检查覆盖率
echo "Checking coverage..."
COVERAGE=$(cat coverage/coverage-summary.json | grep '"total"' | grep -oP '"lines":\{"total":\K[0-9.]+')
if (( $(echo "$COVERAGE < 50" | bc -l) )); then
  echo "Coverage is below 50%!"
  exit 1
fi

echo "All checks passed!"
```

## 5. 质量报告

### 5.1 SonarQube 报告

#### 5.1.1 查看报告
- 登录 SonarQube 服务器
- 选择 StarPivot 项目
- 查看"质量门禁"状态
- 查看"问题"列表
- 查看"覆盖率"报告

#### 5.1.2 报告解读
- **红色**: 质量门禁失败，必须修复

## 6. 质量改进

### 6.1 持续改进计划

#### 6.1.1 短期目标（1-2 个月）
- 代码覆盖率提升到 50%
- 消除所有高危安全漏洞
- 代码重复率降低到 3%

#### 6.1.2 中期目标（3-6 个月）
- 代码覆盖率提升到 70%
- 新增代码覆盖率保持 80% 以上
- 技术债务降低 50%

#### 6.1.3 长期目标（6-12 个月）
- 代码覆盖率保持 80% 以上
- 所有质量指标达到 A 级
- 建立自动化质量监控体系

## 7. 常见问题

### Q1: 质量门禁失败怎么办？
A: 
1. 查看 CI/CD 日志，确定失败原因
2. 修复相关问题
3. 本地验证通过后重新提交

### Q2: 覆盖率不够怎么办？
A:
1. 识别未覆盖的代码
2. 添加相应的测试用例
3. 确保测试覆盖主要逻辑和边界情况

### Q3: SonarQube 扫描失败怎么办？
A:
1. 检查 SonarQube 服务器连接
2. 检查 SONAR_TOKEN 配置
3. 查看扫描日志，定位问题

### Q4: 如何临时绕过质量门禁？
A: 
- 不建议绕过质量门禁
- 如有紧急情况，需要技术负责人批准
- 必须在后续 PR 中修复相关问题
