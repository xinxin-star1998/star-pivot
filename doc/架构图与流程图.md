# StarPivot 架构图与流程图

本文档对 [README 第 3 节「架构图与流程图」](../README.md#3-架构图与流程图) 做补充，提供分层架构图、RBAC 权限模型、前端路由与菜单加载等流程图，便于开发与评审时查阅。

## 1. 分层系统架构图（从客户端到基础设施）

下图按「客户端 → 过滤器链 → 业务模块 → 接口定义 → 框架层 → 依赖管理 → 基础设施」分层，与常见企业级 Spring Boot 架构图风格一致。

```mermaid
flowchart TB
    subgraph Client["客户端层 Client Layer"]
        Web1[Web 后台<br/>star-pivot-ui Vue3]
        Web2[Web 前台]
        App[移动端 APP]
        direction TB
    end

    subgraph Entry["统一入口"]
        Tomcat["Spring Boot 内嵌 Tomcat<br/>端口: 8080  context-path: /api"]
    end

    subgraph AppLayer["应用层 / 过滤器链 Application Layer"]
        F1["CorsFilter 跨域处理"]
        F2["JwtAuthenticationFilter Token 认证"]
        F3["Spring Security 过滤器链"]
        F1 --> F2 --> F3
    end

    subgraph Biz["业务模块层 Business Module Layer"]
        M1["star-pivot-system<br/>系统管理"]
        M2["star-pivot-generator<br/>代码生成"]
        M3["star-pivot-module-xxx<br/>扩展模块"]
    end

    subgraph API["接口 / 控制层 API & Controller"]
        A1["Controller"]
        A2["Service"]
        A3["Mapper / 实体"]
    end

    subgraph Framework["框架层 Framework Layer"]
        FW1["framework-core 核心工具"]
        FW2["framework-log 操作日志"]
        FW3["framework-file 文件 MinIO/OSS"]
        FW4["framework-web Web 抽象"]
        FW5["star-pivot-common 通用聚合"]
        FW6["star-pivot-security 安全与 JWT"]
        FW7["star-pivot-quartz 定时任务"]
        FW8["framework-boot-starter 启动聚合"]
    end

    subgraph Dep["依赖管理层 Dependency Management"]
        BOM["star-pivot-dependencies<br/>版本 BOM"]
    end

    subgraph Infra["基础设施层 Infrastructure Layer"]
        MySQL["MySQL 8.0+<br/>主数据库"]
        Redis["Redis 7.0+<br/>缓存 / JWT 黑名单"]
        MinIO["MinIO / 本地文件<br/>日志与上传"]
    end

    Web1 --> Tomcat
    Web2 --> Tomcat
    App --> Tomcat
    Tomcat --> F1
    F3 --> M1
    F3 --> M2
    F3 --> M3
    M1 --> A1
    M2 --> A1
    M3 --> A1
    A1 --> A2
    A2 --> A3
    A3 --> FW8
    FW8 --> FW5
    FW8 --> FW6
    FW8 --> FW7
    FW5 --> FW1
    FW5 --> FW2
    FW5 --> FW3
    FW5 --> FW4
    FW8 --> BOM
    FW6 --> Redis
    FW5 --> MySQL
    FW3 --> MinIO
```

说明：

- **客户端层**：Web 后台（Vue 3）、Web 前台、移动端等，统一通过 Tomcat 端口 8080、前缀 `/api` 访问。
- **应用层**：请求先经 CORS 配置，再经 `JwtAuthenticationFilter` 解析 Token、校验黑名单，最后经 Spring Security 过滤器链。
- **业务模块层**：`star-pivot-system`（用户/角色/菜单/部门等）、`star-pivot-generator`（代码生成），可扩展 `star-pivot-module-xxx`。
- **接口/控制层**：Controller 暴露 REST API，调用 Service 与 Mapper，对应实体与 DTO。
- **框架层**：核心、日志、文件、Web、common 聚合、security、quartz、boot-starter 等，为业务提供通用能力。
- **依赖管理层**：`star-pivot-dependencies` 统一管理依赖版本。
- **基础设施层**：MySQL 持久化、Redis 缓存与 JWT 黑名单、MinIO/本地文件存储。

## 2. 系统总体架构（概览）

系统采用前后端分离：**star-pivot-ui**（Vue 3）通过 HTTP 访问 **star-pivot-controller**（Spring Boot），后端依赖 **star-pivot-framework** 与 **star-pivot-module**，数据存储为 MySQL、Redis、MinIO/OSS。详见 README 中的「3.1 系统总体架构图」。

## 3. RBAC 权限模型关系图

```mermaid
erDiagram
    SysUser ||--o{ UserRole : 拥有
    SysRole ||--o{ UserRole : 被分配
    SysRole ||--o{ RoleMenu : 关联
    SysMenu ||--o{ RoleMenu : 被授权
    SysUser ||--o{ UserPost : 任职
    SysPost ||--o{ UserPost : 被分配
    SysUser }o--|| SysDept : 所属

    SysUser {
        int id
        string username
        string password
        int dept_id
    }

    SysRole {
        int id
        string role_name
        string role_key
    }

    SysMenu {
        int id
        int parent_id
        string path
        string component
        string perms
    }

    UserRole {
        int user_id
        int role_id
    }

    RoleMenu {
        int role_id
        int menu_id
    }

    SysDept {
        int id
        int parent_id
        string dept_name
    }

    SysPost {
        int id
        string post_name
    }

    UserPost {
        int user_id
        int post_id
    }
```

- **用户 (SysUser)** 通过 **用户-角色 (UserRole)** 关联 **角色 (SysRole)**。
- **角色** 通过 **角色-菜单 (RoleMenu)** 关联 **菜单/权限 (SysMenu)**，菜单的 `perms` 对应按钮级权限。
- 登录后后端返回用户、角色与菜单列表，前端据此生成动态路由与侧栏菜单，并用 `v-auth` 等控制按钮显隐。

## 4. 前端路由与菜单加载流程

```mermaid
flowchart TB
    A[用户打开系统] --> B{本地是否有 token?}
    B -->|否| C[跳转登录页]
    B -->|是| D[进入路由守卫]
    D --> E[调用 getCurrentUser / userMenuTree]
    E --> F{接口是否成功?}
    F -->|否| G[清除 token，跳转登录]
    F -->|是| H[写入用户与菜单到 Store]
    H --> I[根据菜单树生成动态路由]
    I --> J[addRoute 注册动态路由]
    J --> K[渲染侧边栏菜单]
    K --> L[进入目标页面]
    C --> M[登录成功]
    M --> D
```

- 动态路由来源于后端菜单树（`path`、`component` 等），前端在路由守卫中拉取后调用 `router.addRoute()` 注册，再渲染侧边栏。

## 5. 缓存与菜单/字典数据流

```mermaid
flowchart LR
    subgraph 读路径
        A1[请求菜单树] --> A2{Redis 有缓存?}
        A2 -->|是| A3[返回缓存]
        A2 -->|否| A4[查 DB]
        A4 --> A5[写入 Redis]
        A5 --> A3
    end

    subgraph 写路径
        B1[菜单/字典变更] --> B2[更新 DB]
        B2 --> B3["@CacheEvict"]
        B3 --> B4[清除对应缓存]
    end
```

- 菜单树：`menuTree`，key 如 `'all'`；字典：`dictData`，key 为 `dictType`。变更时使用 `@CacheEvict(allEntries = true)` 或按 key 驱逐。
