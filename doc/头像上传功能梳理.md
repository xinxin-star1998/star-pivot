# 头像上传功能梳理

## 一、流程概览

### 1. 前端

| 位置 | 组件/接口 | 说明 |
|------|------------|------|
| **组件** | `star-pivot-ui/src/components/core/media/art-avatar-upload/index.vue` | 头像上传组件，支持 v-model(avatar)、userId、autoUpload、usePresignedUrl |
| **接口** | `fetchUploadAvatar(FormData)` → `POST /api/avatar/upload` | 上传：file、userId、usePresignedUrl |
| | `fetchDeleteAvatar(userId)` → `DELETE /api/avatar/delete` | 删除头像 |
| | `fetchGetAvatarPresignedUrl(filePath)` → `GET /api/avatar/presigned-url` | 获取临时访问链接 |
| **使用场景** | 系统管理 → 用户管理（新增/编辑用户弹窗） | `user-dialog.vue`，编辑时 autoUpload=true |
| | 个人中心 | `user-center/index.vue`，编辑时自动上传 |

**流程简述：**

- **选图**：点击头像 → 选文件 → 若 `autoUpload=true` 立即调 `POST /api/avatar/upload`，拿到 `avatarUrl` 后更新 `v-model`（即 `formData.avatar` / `form.avatar`）。
- **新增用户**：`autoUpload=false`，先保存用户拿到 `userId`，再调用 `avatarUploadRef.uploadImageToServer()` 上传头像；保存用户时 `formData` 中已包含 `avatar`（上传返回的 URL）。
- **保存用户**：`fetchUpdateUser` / `fetchAddUser` 会把 `avatar` 随表单提交，后端 `SysUserServiceImpl.updateUser` 通过 `BeanUtils.copyProperties` 将 `avatar` 写入 `sys_user.avatar`。

### 2. 后端

| 模块 | 说明 |
|------|------|
| **控制器** | `star-pivot-controller/.../AvatarController.java`，路径前缀 `/avatar` |
| **接口** | `POST /avatar/upload`：上传头像，返回 `avatarUrl`（永久 URL 或临时 Presigned URL） |
| | `GET /avatar/presigned-url?filePath=xxx`：根据对象路径获取临时访问链接 |
| | `DELETE /avatar/delete?userId=xxx`：按用户删除头像 |
| **权限** | 仅允许操作本人头像，或具备 `system:user:edit` 的管理员（且不能操作超管账号） |
| **存储** | 当前**仅使用 `OssUtil`**（阿里云 OSS）；`MinioUtil` 已注入但未使用 |
| **用户表** | `sys_user.avatar` 在用户新增/更新时由 `SysUserServiceImpl` 正常落库 |

存储逻辑：

- 对象路径格式：`avatar/{userId}.{suffix}`（如 `avatar/1.png`）。
- OSS：先上传新文件，再删除该用户下其它旧头像。
- 上传接口可根据参数返回「永久 URL」或「临时 Presigned URL」。

## 二、已发现的问题与修复

### 问题 1：用户管理弹窗使用临时链接存库，过期后头像失效（已修复）

**现象**：在「系统管理 → 用户管理」中编辑用户并上传头像时，若上传接口只返回临时签名 URL 并写入库，过期后头像无法访问。

**修复**：上传接口在 `usePresignedUrl=true` 时**同时返回** `avatarUrl`（永久 URL，用于存库）和 `presignedUrl`（临时链接，仅用于前端展示）；前端存库只用 `avatarUrl`，展示用 `presignedUrl` 或 `avatarUrl`。这样既避免存库临时链接过期，又能在私有桶下正常显示。

### 问题 1.1：编辑用户时已有头像 403 Forbidden（已修复）

**现象**：OSS 桶为私有时，库中存的是永久 URL（如 `https://star-pivot.oss-cn-beijing.aliyuncs.com/avatar/113.webp`），前端直接用该 URL 作为 `<img src>` 请求会返回 **403 Forbidden**，头像不显示。

**修复**：用户管理弹窗中保留 `use-presigned-url`。组件在展示时若发现是 OSS 地址，会先根据 URL 解析出对象路径，请求后端 `GET /api/avatar/presigned-url?filePath=avatar/113.webp` 获取临时可访问链接，再用该链接作为图片地址展示，从而在私有桶下正常显示；存库仍为永久 URL。

### 问题 2：后端只使用 OSS，MinIO 未参与头像逻辑

**现象**：配置中已有 `minio.bucket-name`（如 `star-pivot-avatar`），但 `AvatarController` 中上传、删除、获取临时链接均只调用 `OssUtil`，`MinioUtil` 虽已注入却未使用。若希望本地或部分环境用 MinIO 存头像，当前无法生效。

**建议**（可选后续改造）：

- 增加配置项，例如：`star-pivot.file.avatar-storage=oss|minio`，默认 `oss`。
- 在 `AvatarController` 中根据该配置在 `OssUtil` 与 `MinioUtil` 之间切换（上传、删除、presigned-url 三者一致使用同一实现）。
- 若存在「仅配 MinIO、不配 OSS」的环境，需要配合 `@ConditionalOnProperty` 或 `@Autowired(required = false)` 等，避免 OSS 必选导致启动失败。

---

## 三、小结

- 头像上传、保存到 `sys_user.avatar`、个人中心/用户管理两处使用流程已打通；问题集中在「存库 URL 类型」和「存储实现只走 OSS」。
- **已修复**：用户管理弹窗改为使用永久 URL 存库，避免临时链接过期后头像失效。
- **待选做**：按配置在 OSS 与 MinIO 之间切换头像存储，并处理好仅 MinIO 环境的配置与 Bean 注入。
