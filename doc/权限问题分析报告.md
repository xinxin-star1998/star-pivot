# 权限问题分析报告

## 一、数据权限优先级错误 ⚠️ **严重**

### 问题描述
在 `DataScopeService.java` 中，数据权限优先级映射存在错误。

### 问题位置
`star-pivot-system/src/main/java/com/star/pivot/system/utils/DataScopeService.java` 第44-51行

### 问题详情
根据常量定义（`Constants.java`）：
- `DEPT = "3"`（本部门）
- `DEPT_AND_CHILD = "4"`（本部门及以下）

但代码中的优先级映射为：
```java
Constants.DataScope.DEPT_AND_CHILD, 3,  // 优先级3
Constants.DataScope.DEPT, 4,            // 优先级4
```

**错误分析**：
- 优先级数字越小，权限越大
- 当前代码中 `DEPT_AND_CHILD`（本部门及以下）的优先级是3，`DEPT`（本部门）的优先级是4
- 这意味着"本部门及以下"的权限比"本部门"更大，这是**逻辑错误**
- 应该是"本部门"的权限范围更小，优先级应该更高（数字更小）

### 正确逻辑
"本部门"的权限范围 < "本部门及以下"的权限范围，所以：
- `DEPT`（本部门）应该优先级更高（数字更小，如3）
- `DEPT_AND_CHILD`（本部门及以下）应该优先级更低（数字更大，如4）

### 修复建议
```java
/** 数据权限优先级：1(全部) > 2(自定义) > 3(本部门) > 4(本部门及以下) > 5(仅本人) */
private static final Map<String, Integer> SCOPE_PRIORITY = Map.of(
        Constants.DataScope.ALL, 1,
        Constants.DataScope.CUSTOM, 2,
        Constants.DataScope.DEPT, 3,              // 修正：本部门优先级3
        Constants.DataScope.DEPT_AND_CHILD, 4,    // 修正：本部门及以下优先级4
        Constants.DataScope.SELF, 5
);
```

同时需要更新注释：
```java
/**
 * 获取当前登录用户的数据权限。
 * <p>
 * 流程：未登录 → 无数据；超级管理员（userId=1）→ 全部数据；
 * 否则按角色数据范围取最大权限（1 > 2 > 3 > 4 > 5），并汇总涉及的部门 ID。
 * </p>
 */
```

## 二、SQL注入风险 ⚠️ **严重**

### 问题描述
在 MyBatis XML 映射文件中使用了 `${}` 直接拼接 SQL，存在潜在的 SQL 注入风险。

### 问题位置
`star-pivot-system/src/main/resources/mapper/SysUserMapper.xml` 第73行

### 问题详情
```xml
AND ${param.dataScope.sqlFilter}
```

虽然 `DataScopeService` 中生成的 SQL 片段使用了硬编码的常量，理论上相对安全，但使用 `${}` 直接拼接仍然存在风险：
1. 如果未来代码修改，可能引入用户输入
2. 不符合 MyBatis 最佳实践（应尽量使用 `#{}`）
3. 代码审查时容易被标记为安全问题

### 修复建议
考虑使用 MyBatis 的 `<choose>` 标签来动态构建 SQL，或者确保 `sqlFilter` 的值完全可控且经过验证。

**方案1：使用 `<choose>` 标签**
```xml
<choose>
    <when test="param.dataScope.sqlFilter == '1=1'">
        <!-- 全部数据，不需要额外条件 -->
    </when>
    <when test="param.dataScope.sqlFilter == '1=0'">
        AND 1=0
    </when>
    <when test="param.dataScope.deptIds != null and param.dataScope.deptIds.size() > 0">
        AND u.dept_id IN
        <foreach collection="param.dataScope.deptIds" item="deptId" open="(" separator="," close=")">
            #{deptId}
        </foreach>
    </when>
    <when test="param.userDeptId != null">
        AND u.dept_id = #{param.userDeptId}
    </when>
    <when test="param.userId != null">
        AND u.create_by = #{param.userId}
    </when>
</choose>
```

**方案2：在 DataScope 中添加权限类型枚举**
在 `DataScope` 中添加 `dataScopeType` 字段，在 XML 中根据类型判断，避免直接拼接 SQL。

## 四、其他查询方法缺少数据权限 ⚠️ **功能缺失**

### 问题描述
`getUserListByRoleId` 和 `unallocatedList` 方法没有应用数据权限过滤。

### 问题位置
1. `star-pivot-system/src/main/java/com/star/pivot/system/service/impl/SysUserServiceImpl.java`
   - 第278行：`getUserListByRoleId()`
   - 第293行：`unallocatedList()`

2. `star-pivot-system/src/main/resources/mapper/SysUserMapper.xml`
   - 第95-113行：`getUserListByRoleId` SQL
   - 第114-131行：`unallocatedList` SQL

### 问题详情
这两个方法用于角色分配用户功能，但没有应用数据权限过滤，可能导致：
- 用户可以看到超出其数据权限范围的用户列表
- 违反了数据权限控制的基本原则

### 修复建议
在这两个方法中也应用数据权限过滤，类似于 `pageList()` 方法的实现：

```java
@Override
public PageResponse<SysUser> getUserListByRoleId(AssignUserReqBo assignUserReqBo) {
    PageResponse<SysUser> pageResponse = new PageResponse<>();
    Page<SysUser> page = new Page<>(assignUserReqBo.getPageNum(), assignUserReqBo.getPageSize());
    
    // 获取当前用户数据权限
    DataScope dataScope = dataScopeService.getCurrentUserDataScope();
    Map<String, Object> param = new HashMap<>();
    param.put("dataScope", dataScope);
    param.put("deptIds", dataScope.getDeptIds());
    param.put("userDeptId", dataScope.getUserDeptId()); // 需要先修复问题三
    param.put("userId", dataScope.getUserId());
    param.put("assignUserReqBo", assignUserReqBo);
    
    IPage<SysUser> pageList = sysUserMapper.getUserListByRoleId(page, param);
    // ... 后续处理
}
```

同时在对应的 XML 中添加数据权限过滤条件。

## 五、表别名硬编码问题 ⚠️ **可维护性**

### 问题描述
`DataScopeService` 中的 SQL 片段硬编码了表别名 `u`。

### 问题位置
`star-pivot-system/src/main/java/com/star/pivot/system/utils/DataScopeService.java`
- 第40行：`PLACEHOLDER_USER_DEPT_ID = "u.dept_id = #{param.userDeptId}"`
- 第42行：`PLACEHOLDER_USER_ID = "u.create_by = #{param.userId}"`
- 第203行：`return "u.dept_id IN (...)"`

### 问题详情
如果其他查询使用了不同的表别名（如 `t`、`user` 等），这些 SQL 片段将无法使用，限制了代码的复用性。

### 修复建议
**方案1：将表别名作为参数传入**
```java
public DataScope getCurrentUserDataScope(String tableAlias) {
    // ... 使用 tableAlias 构建 SQL
}
```

**方案2：在 DataScope 中存储不带别名的字段名**
```java
public class DataScope {
    private String sqlFilter;  // 如：dept_id IN (1,2,3)
    private String tableAlias; // 如：u
    // 使用时拼接：tableAlias + "." + sqlFilter
}
```

**方案3：使用占位符，在 XML 中替换**
在 XML 中使用 `<bind>` 标签或字符串替换来处理表别名。

## 六、数据权限范围计算逻辑问题 ⚠️ **逻辑**

### 问题描述
在 `calculateDataScope()` 方法中，当多个角色具有相同优先级时，只处理了 `CUSTOM` 类型的累加，其他类型没有处理。

### 问题位置
`star-pivot-system/src/main/java/com/star/pivot/system/utils/DataScopeService.java` 第105-129行

### 问题详情
```java
if (isHigherPriority(scope, highestScope)) {
    highestScope = scope;
    deptIdSet.clear();  // 清空之前的部门ID
    processScope(scope, role.getRoleId(), userDeptId, deptIdSet);
} else if (scope.equals(highestScope) && Constants.DataScope.CUSTOM.equals(scope)) {
    // 自定义权限：累加多个角色的部门ID
    addCustomDeptIds(role.getRoleId(), deptIdSet);
}
```

**问题**：
- 如果用户有两个角色，都是 `DEPT_AND_CHILD` 类型，第二个角色的处理会被忽略
- 虽然 `DEPT_AND_CHILD` 和 `DEPT` 类型基于用户部门计算，不需要累加，但逻辑上应该明确说明

### 修复建议
添加注释说明，或者优化逻辑使其更清晰：
```java
// 比较优先级，只处理更高优先级的权限
if (isHigherPriority(scope, highestScope)) {
    highestScope = scope;
    deptIdSet.clear();
    processScope(scope, role.getRoleId(), userDeptId, deptIdSet);
} else if (scope.equals(highestScope)) {
    // 相同优先级的权限处理
    if (Constants.DataScope.CUSTOM.equals(scope)) {
        // 自定义权限：累加多个角色的部门ID
        addCustomDeptIds(role.getRoleId(), deptIdSet);
    }
    // 其他类型（DEPT、DEPT_AND_CHILD、SELF）基于用户部门或用户ID计算，不需要累加
}
```
